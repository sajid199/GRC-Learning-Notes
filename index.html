Post Installation Checks (Pre-Checks)

▼
ℹ️
These checks are done right after GRC Access Control is installed to ensure all required components and settings are in place before going live.
Step 1

Check Required Software Components

Verify that all necessary GRC software components are installed in both systems.

SystemRequired ComponentDescription
GRC SystemGRCFND_ACore GRC Foundation – Access Control
Plug-in System (ECC)GRCPINWGRC Plug-in for the backend ECC system
Login to GRC System → Top Menu → System → Installed Products
Image 1 — Installed Products: verify GRCFND_A on GRC system and GRCPINW on plug-in system
Step 2

Verify BC Set Activation (SCPR20)

📘
BC Set = Business Configuration Set — SAP pre-defined default/standard configuration values for GRC components.

Use T-code SCPR20 and confirm the following BC Sets are activated:

BC Set NameGRC Module
ACCESS_REQUESTARM – Access Request Management
ROLE_MGMTBRM – Business Role Management
RULESETARA – Access Risk Analysis
SPMEAM – Emergency Access Management
Image 2 — SCPR20: BC Set activation list for GRC modules
Step 3

Verify Services Activation (SICF)

Use T-code SICF to confirm all services related to GRAC and NWBC are activated. These services enable browser-based GRC access via NetWeaver Business Client.

Step 4

Activate GRC Applications in Client

SPRO → GRC → General Settings → Activate Applications in Client

Ensure GRC-AC (Access Control) is set to Active.

Image 3 — Activate Applications in Client: GRC-AC should show Active status
Step 5

Perform Automatic Workflow Customizing

SPRO → GRC → General Settings → Workflow → Perform Automatic Workflow Customizing
✅
All entries must show GREEN. If any item is red or yellow, contact the BASIS team immediately.
Image 4 — Automatic Workflow Customizing: everything must be GREEN
Step 6

Perform Task-Specific Workflow Customizing

SPRO → GRC → General Settings → Workflow → Perform Task-Specific Customizing
⚠️
Every task must be defined as either Background Task or General Task. If any task entry is empty, set it as General Task.

Connector Configuration

▼
⚠️
Prerequisite: You need a User ID in the Plug-in system (ECC) for RFC connectivity. The user must be of type SYSTEM or COMMUNICATION and should have the necessary RFC authorizations assigned. If a defined role is not available, you can assign SAP_ALL — there is no risk because a SYSTEM/COMMUNICATION user cannot be used to log in interactively even if someone knows the password.
🔌
Common path for Steps 1–3:
SPRO → GRC → Common Component Settings → Integration Framework
Step 1

Create Connector — SM59 (RFC Setup)

Set up the RFC (Remote Function Call) connection between the Plug-in system (ECC) and the GRC system.

In SM59, select ABAP Connection and click Create.

Step 2

Maintain Connectors and Connection Types

This step has 5 sub-steps (a through e):

  • a) Define Connectors — The Source Connector and Logical Port must match the Target Connector name (the same SM59 RFC name).
  • b) Define Subsequent Connector — Only required for JAVA systems. Not needed for ABAP systems. This is typically done by the BASIS team.
  • c) Define Connector Groups — Create a connector group if needed. This allows you to manage configuration by group instead of system-by-system. For ARA and ARM, using connector groups is mandatory.
  • d) Assign Connector Groups to Group Types — Set your connector group status as Logical Group for Access Control (AC). For Process Control (PC), select Automated Monitoring Framework.
  • e) Assign Connectors to Connector Groups — Add the RFC connector to the appropriate group (e.g., ECC, BW, SD, Finance can all be placed under one group).

    Sync Jobs

    ▼
    🔄
    Purpose: Sync Jobs are used to pull data from the Plug-in (ECC) system into the GRC system. The ECC data is not created in GRC — it is identified (recognized) in GRC by syncing it. Without sync, GRC cannot see ECC users, roles, or authorization objects.
    SPRO → IMG Reference → GRC → Access Control → Sync Jobs
    Image 23 — Sync Jobs configuration in SPRO
    7 Types

    Types of Sync Jobs

    #Sync JobWhat It SyncsSystem
    1 Authorization Sync T-codes, Authorization Objects, SU24 data ABAP only (not JAVA)
    2 Repository Object Sync Users, Roles, and Profiles ABAP + JAVA
    3 Action Usage Sync T-code usage data ABAP
    4 Role Usage Sync Role assignment to users ABAP
    5 Fire Fighter Log Sync FF ID usage logs from plug-in EAM
    6 Fire Fighter Workflow Sync FF workflow for FF ID Controller review EAM / ARM
    7 Fetch IDM Schema Sync All Portal/JAVA system data JAVA only (IDM = Identity Management)
    Mandatory

    Mandatory Jobs by System Type

    System TypeMandatory Sync Jobs
    ABAP SystemJob 1 (Authorization) + Job 2 (Repository Object)
    JAVA SystemJob 2 (Repository Object) + Job 7 (Fetch IDM Schema)
    Component

    Sync Jobs by GRC Component

    GRC ComponentRequired Sync JobsNote
    ARA (Access Risk Analysis)1, 2, 7Job 7 only for JAVA/Portal systems
    EAM (Emergency Access Mgmt)5, 6Firefighter logs and workflow
    Security Audit Reports3, 4Usage tracking
    ARM (Access Request Mgmt)6FF workflow only
    How To

    How to Execute Sync Jobs

    🚫
    Never execute sync jobs directly in foreground! Always run them in Background mode to avoid GRC system performance issues.

    Go to the program/job and choose Execute in Background. Use T-code SM37 to monitor background job status.

    Image 24 — Sync Job: always choose Execute in Background, not foreground
    Image 25 — Repository Object Sync: Full Sync vs Incremental Mode options
    📋
    Repository Object Sync runs in 2 modes:
    • Full Sync — Complete data load. Run the first time and once a month thereafter.
    • Incremental Mode — Only delta (changed) data. Run daily from the 2nd time onwards.
    Image 26 — SM37: Background Job Monitor — check sync job status here
    Troubleshoot

    If a Sync Job Fails — Common Root Causes

    🚨
    When a sync job fails in SM37, check these 4 root causes in order:
    1
    RFC Connection Problem
    Go to SM59, open the RFC connection and click Test Connection. A failed ping here is the most common cause.
    2
    Post Installation Step 5 Incomplete
    Check that Perform Automatic Workflow Customizing was completed and shows all GREEN.
    3
    RFC User Missing Authorizations
    Ensure the RFC user in the ECC plug-in system has proper RFC authorizations. If needed, temporarily assign SAP_ALL to test.
    4
    Connector Configuration — Integration Scenarios
    Verify all 5 integration scenarios (AUTH, PROV, ROLMG, SUPMG, PSS) are correctly maintained under Integration Framework.
